## 核心认知

### 1. RAG 多轮对话的修复
之前 `gradio_app.py` 的 `respond(message, history)` 函数接收了 `history` 参数但没有使用，导致每次问答都是独立的，无法追问。

**修复方式**：
- 在 prompt 模板中加入 `{chat_history}` 占位符
- 新增 `format_chat_history()` 函数，把 Gradio 的 `[[用户, AI], ...]` 格式转为可读字符串
- 只保留最近 4 轮历史，避免 prompt 过长
- 检索仍然只基于当前 `message`，不用历史去检索（防止检索到不相关文档）

### 2. 发现向量库重复数据问题
评估脚本跑出来发现：每个问题的 top-3 结果都是同一份文档。根本原因：Chroma 的 `add_documents()` 和 `from_documents()` 不会自动去重。开发过程中多个脚本（如 `langchain_rag.py`、`gradio_app.py` 的自动初始化逻辑）在启动时都检测不到向量库或检测逻辑不完善，各自调用了 `from_documents()` 追加写入，导致同一个 collection 里累积了重复的 chunk。

**修复**：用 `manage_docs.py rebuild` 先删除整个向量库目录，再从 `data/` 重新构建。重建后 top-3 变成了不同文档的不同片段，质量明显改善。

**经验教训**：这是典型的生产问题——向量库操作前要先想清楚再动手。企业做法：维护文档 hash 表去重，或用支持 upsert 的向量库，或在写入前检查 collection 是否为空。

### 3. 企业级 RAG 的差距
对比真实的客户交付场景，目前的 RAG 系统还缺少：
- PDF 支持（目前只支持 .md / .txt）
- Reranker 精排（目前是向量检索直接取 top-3）
- 回答质量评估（目前是人工看关键词是否命中）
- 并发请求支持（Gradio 是单进程）
- 日志/监控（生产环境需要记录每次请求的检索质量和响应时间）
