### K8s 核心概念串讲（问答互动式复习）

1. **K8s 解决什么问题**：不是"容器编排"四个字，而是"怎么让一堆应用在多台机器上可靠地跑起来"。Docker 管单机（怎么打包跑起来），K8s 管集群（怎么可靠地跑、挂了自动拉起、扩容、迁移）。

2. **Pod vs Container**：Pod 提供两样 Container 没有的东西——共享网络命名空间（同 IP，localhost 通信）+ 共享存储卷。Sidecar 模式是核心价值。

3. **Deployment → ReplicaSet → Pod**：职责分离。Deployment 管版本，ReplicaSet 管数量。滚动更新靠新旧两个 ReplicaSet 共存，全程维持期望副本数零停机。

4. **Service 本质**：不是进程，是一个声明对象。kube-proxy + iptables 才是真正转发的。Endpoints 是"通讯录"，selector 选不中 = Endpoints 为空 = 空壳。

5. **三种 Probe**：startupProbe → "启动完成了吗？"（不通过杀 Pod）；livenessProbe → "还活着吗？"（不通过杀 Pod 重启）；readinessProbe → "能接客了吗？"（不通过从 Endpoints 摘掉，不杀）。

6. **requests vs limits**：requests 是调度依据（选节点），limits 是运行时上限（cgroup 限制）。超 memory → OOMKilled；超 CPU 不会被杀，只是限速（throttle）。

7. **GPU 管理**：需要 Device Plugin 注册为 `nvidia.com/gpu`。三种共享方案：MPS（无隔离）/ vGPU（软件隔离）/ MIG（硬件隔离，仅 A100+）。GPU 显存和系统内存是两套独立系统，K8s 只管系统内存不管 GPU 显存。

8. **声明式 API**：apply 幂等性 + reconcile 循环。你告诉它"要什么"，它负责"怎么做到"。

### 易遗忘点补充

- GPU 显存（VRAM）和系统内存是两套独立的内存系统，K8s 只管系统内存
- 排查重启问题的关键：`kubectl logs --previous` 看崩溃前的日志
- Exit Code 137 = OOMKilled，1 = 程序报错，143 = 正常终止
- Service selector 配错不会报错，只是静默地 Endpoint 为空，很难排查

K8s 的名词容易混，但用问答的方式串一遍，比单独背概念有效得多。三种 Probe 的设计体现了对"健康"的多层次理解：能启动 → 还活着 → 能接客。GPU 显存和系统内存的独立性是容易忽略的点，联旌客户部署大模型时，显存爆掉是最常见的故障之一，但 K8s 不会记录 OOMKill 事件。
