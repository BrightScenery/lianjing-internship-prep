### 学了什么

1. **Agent 通用框架 = 循环结构 + 可替换工具**
   - 核心循环（while + max_turns + messages 累积）从昨天到今天完全没变
   - 只是把"模拟 GPU 数据工具"换成了"真实搜索引擎工具"
   - 这就是 Agent 的可扩展性：加新工具不改循环结构

2. **工具设计原则**
   - 工具描述要清晰说明"做什么 + 什么时候用"，模型才知道什么时候调
   - 搜索结果要截断（2000字上限），防止上下文窗口被塞满
   - 工具返回值影响 Agent 行为：空数据会导致模型反复尝试

3. **多轮对话 vs 单轮对话**
   - 单轮：每次重新初始化 messages（不记得之前说了什么）
   - 多轮：messages 一直累积（模型有完整记忆）

4. **实战效果**
   - 测试1（Kubernetes 是什么）：1轮搜索 → 总结回答，高效
   - 测试2（进一步介绍架构）：模型自主中英双语搜索，3轮完成
   - 测试3（GPU MIG 技术）：3轮搜索，回答专业详
   - 模型会自己选关键词、换语言搜索、逐步深入——这就是 Agent 的价值

### 容易踩的坑

1. **ddgs 包取代了 duckduckgo_search**：旧包已废弃，pip install ddgs
2. **Windows 终端中文编码**：in-line python 命令中文乱码，用脚本文件解决
3. **交互模式 vs 测试模式**：交互模式效果远好于测试模式，因为多轮对话让模型能积累搜索结果
4. **Agent 行为取决于工具返回值**：工具返回空数据，模型会反复调用直到信息够了

## 💡 今日感悟
Agent 的核心不是"多复杂的代码"，而是一个简单的 while 循环。真正决定 Agent 能力上限的是它能调用什么工具。给 Agent 搜索引擎，它就是知识助手；给 GPU 监控工具，它就是运维助手——同一个循环结构，换工具就是换能力。这和联旌的 EaaS 平台思路一致：底层是统一的容器+调度架构，上层装不同的应用就是不同的服务。
