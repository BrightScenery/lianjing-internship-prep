### 学了什么

1. **Agent 的本质**
   - Function Calling 是"单次调用"：模型调一次工具 → 拿结果 → 回答
   - Agent 是"循环自主决策"：模型在 while 循环中反复思考 → 调工具 → 观察 → 再思考，直到信息够了才回答
   - Agent = 大模型 + 工具 + 自主决策循环

2. **ReAct 范式（Reasoning + Acting）**
   ```
   Thought（思考）→ Action（调用工具）→ Observation（观察结果）→ 重复 → Final Answer
   ```
   每一轮都记录在 messages 中，所以模型知道"我之前查过什么"

3. **核心代码结构**
   - while 循环 + max_turns 上限
   - 每轮都传 tools 参数
   - 工具结果追加到 messages 历史
   - 模型没有 tool_calls 时 → 退出循环

4. **实战效果**
   - 测试1（GPU-1 慢）：模型自主决定先查使用率 → 再查进程 → 给出诊断，3轮完成
   - 测试2（集群状况）：模型先查告警 → 同时调3个工具 → 查进程 → 出完整报告，4轮完成
   - 模型自己决定了查什么、查几个、什么时候停

### 容易踩的坑

1. **必须设 max_turns**：防止模型一直调工具不停下来
2. **每轮都要传 tools**：否则模型"忘记"可用工具
3. **不能丢 messages 历史**：模型需要知道之前查过什么
4. **system prompt 要明确**：告诉 Agent 工作流程，否则可能调错工具或过早结束

## 💡 今日感悟
Function Calling 是 Agent 的基础组件，Agent 是 Function Calling 的循环升级版。前者是"帮你点一个菜"，后者是"帮你规划一整趟旅行"。联旌的 GPU 调度、故障排查等运维场景，底层都可以用 Agent 范式来实现——模型当"指挥官"，工具当"执行者"。
