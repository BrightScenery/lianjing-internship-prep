### 学了什么

1. **Function Calling 的本质**
   - 模型不直接执行函数，它只返回"需要调用哪个函数 + 参数是什么"
   - 你在代码中执行函数，把结果以 `role: "tool"` 返回给模型
   - 模型根据工具结果生成最终自然语言回答

2. **6步流程**
   ```
   用户提问 → 模型分析（传入tools） → 模型返回tool_calls
   → 你执行函数 → 把结果返回给模型 → 模型生成最终回答
   ```

3. **关键细节**
   - tools 描述三要素：name + description + parameters，缺一不可
   - 多轮对话中每轮都要传 tools 参数，否则模型"忘记"可用工具
   - tool_call_id 用于匹配"这个结果对应哪个工具调用"
   - 工具执行失败要用 try/except 捕获，把错误信息返回给模型

4. **多轮对话实战**
   - 用户说"那 GPU-1 呢？"，模型正确理解上下文，自动调用 get_gpu_status
   - 保持 messages 历史是关键，模型需要上文才能正确解析模糊意图

### 问答记录

**Q1：如果用户问"你好"（不涉及任何工具），模型会怎么做？**
A：直接回答。因为模型会拿用户问题和工具描述做匹配，发现没有匹配的工具，就用自己的知识回答。

**Q2：tools 的 description 字段重要吗？如果写得很模糊会怎样？**
A：非常重要。模型判断是否调用工具、调用哪个工具，主要依据就是 name + description + parameters 这三个描述字段。如果描述模糊，模型可能找不到正确的函数。函数命名也要见名知意，`get_weather` 比 `func1` 好一万倍。

**Q3：为什么工具结果要用 `role: "tool"` 而不是 `role: "user"` 返回？**
A：`role: "tool"` 让模型知道这是工具返回的数据，不是用户在说话，从而更精准地组织回答。另外 `tool` 消息里有 `tool_call_id`，模型用这个来匹配"这个结果对应之前哪个工具调用"。

### 容易踩的坑

1. **模型返回的参数可能不符合预期**：比如 `gpu_id` 定义是字符串，但用户问"所有 GPU"，模型可能返回 `"all"` 或 `"*"`，函数需要处理边界情况
2. **工具执行失败**：不能直接把异常抛给用户，要把错误信息作为 tool 结果返回给模型，让模型用自然语言告知
3. **多轮对话中 tools 参数**：每轮调用都必须传入 tools，否则模型"忘记"了有哪些工具可用
