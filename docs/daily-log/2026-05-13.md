## 核心认知

### 1. RAG vs TF-IDF
- TF-IDF 只认字面，不懂语义——"轻量产品"匹配不到"轻量版"
- RAG 用 Embedding 模型把文字变成向量，语义相近的内容向量距离也近
- **RAG 的检索质量几乎完全取决于 Embedding 模型的选择**

### 2. Embedding 模型踩坑
- `nomic-embed-text`：英文优化，中文效果极差 → 换了 `bge-m3` 后检索准确率天差地别
- ChromaDB 对不同 Embedding 模型的维度要求不同，bge-m3 是 1024 维
- 不同模型生成的向量不兼容，换模型必须重建索引

### 3. Prompt 工程
- 同样一段代码、同一份资料，Prompt 改几个字，回答从"资料中没有"变成"是的，支持"
- Prompt 的约束力比想象强得多：模型本身知道 docker=容器，但 Prompt 说"根据资料回答"，它就严格按资料来
- 加"结合自身知识" + "简短回答"后，回答质量大幅提升

### 4. 轻薄本的局限
- 本地跑 LLM 确实吃 CPU，4B 模型就能让 CPU 飙到 30%+
- 实习后模型跑在公司 GPU 服务器上，个人电脑只需发请求
- 第5周切换到云端 API 后就不存在这个问题
