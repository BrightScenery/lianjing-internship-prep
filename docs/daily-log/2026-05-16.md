## 核心认知

### 1. Gradio ChatInterface 的本质
Gradio 不需要写任何 HTML/CSS/JS。只需要定义一个 Python 函数 `respond(message, history)`，它自动生成完整的聊天界面。用户点击发送时，Gradio 调用这个函数，把返回值显示在聊天窗口。这和 Linux 里学过的思想一样——把复杂操作封装成简单接口。

### 2. LCEL 链中 itemgetter 的作用
```python
rag_chain = (
    {"context": itemgetter("question") | retriever, "question": itemgetter("question")}
    | prompt | llm | StrOutputParser()
)
```
`itemgetter("question")` 从输入字典 `{"question": "..."}` 中提取问题字符串。`| retriever` 将其传给检索器，返回 3 个相关文档赋值给 `context`。这行等价于手写版的"先检索，再拼 prompt，再调 LLM"。

### 3. Docker 打包的意义
在腾讯云服务器上 `docker build` 镜像 → `docker save` 导出 .tar → scp 下载到本地 → WSL2 `docker load` → `docker run` 直接运行，不需要在本地安装任何 Python 依赖。完整验证了"Build Once, Run Anywhere"——镜像包含 Python 环境、pip 包、代码、资料，在任何有 Docker 的地方行为一致。

这是**CI/CD 交付链路的基础思想**：真正的 CI/CD 是用工具（如 GitHub Actions）在代码 push 后自动构建镜像并部署，而我们手动走的"本地→云服务器→本地"流程就是验证同一件事——镜像在任何环境下行为一致。实习时会接触到自动化 CI/CD 工具。

### 4. Dockerfile 关键配置
- `ENV GRADIO_SERVER_NAME=0.0.0.0` — Gradio 默认只监听 127.0.0.1（容器内部），必须设 0.0.0.0 才能从外部访问
- `EXPOSE 7860` — 声明端口（仅文档作用，实际需要 `docker run -p` 做映射）
- `RUN pip install --no-cache-dir` — 不缓存 pip 下载文件，减小镜像体积

## 反思
- 先本地跑通代码，再云服务器构建镜像，再下载到本地验证——这个流程是标准的 CI/CD 交付链路，实习时大概率会用到
