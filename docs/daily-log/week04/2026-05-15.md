## 核心认知

### 1. LangChain 本质是将已会做的事封装成通用接口
手写 RAG 的每一行代码（加载文档→分块→向量化→检索→拼prompt→调LLM）都已经写过一遍了。
LangChain 做的事情就是把这些步骤抽象成组件：
- `DirectoryLoader` = 手写的 `glob.glob()` + `open().read()`
- `RecursiveCharacterTextSplitter` = 手写的 `chunk_text()`，但更智能（按段落→换行→空格递归切，不是硬切）
- `OllamaEmbeddings` = 手写的 `ollama.embed()`
- `Chroma.from_documents()` = 遍历 chunks → 调 embedding → 存入 ChromaDB，一行搞定
- `ChatPromptTemplate` = 手写的 f-string 拼 prompt
- `ChatTongyi` = 通义千问的 LangChain 内置集成，不用自己写包装类

**关键理解**：框架不是魔法，它只是通用化、标准化已经会做的操作。先手写再学框架的路径是对的——因为框架的每一行封装了什么，都能对应到自己写过的代码上。

### 2. LCEL（LangChain Expression Language）
新版 LangChain 用 `|` 管道符串组件，等价于手写的"第1步→第2步→第3步"。

```python
answer_chain = (
    {"context": itemgetter("input") | retriever, "input": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)
```

这个 `|` 和 Linux 的管道符是一个概念：前一个组件的输出是后一个组件的输入。

### 3. 分块策略优化
- 原来 chunk_size=400 太小，一个知识点被切成两半，检索时只拿到"半截信息"
- 改为 chunk_size=800 + overlap=150，一个完整知识点更可能落在同一个 chunk 里
- 这对应企业中的调优思路：RAG 回答质量不好，先排查检索环节，再看分块策略

### 4. 踩坑
- **版本兼容**：LangChain 1.2.x 已移除 `langchain.chains` 模块，旧教程的 `create_retrieval_chain` 不可用，必须用 LCEL 方式重写
- **Windows 编码**：`TextLoader` 默认用系统编码（GBK）读文件，但 `.md` 文件是 UTF-8，必须用 `lambda p: TextLoader(p, encoding="utf-8")` 指定
- **本地模型跑不动**：qwen3.5:4b（3.4GB）+ bge-m3（1.2GB）同时在轻薄本上推理，CPU 算力不够。这也验证了之前的认知：生产环境模型必须跑在 GPU 服务器上，个人电脑只发请求
- **Prompt 模板双花括号陷阱**：`{{context}}` 是转义，模板不会填值，必须用单花括号 `{context}`
- **API Key 安全**：不能明文写在代码里，优先读环境变量 `DASHSCOPE_API_KEY`，没有再手动输入
- **langchain-dashscope 包名**：正确的做法是 `pip install langchain-community`，`ChatTongyi` 已经内置在其中，不需要单独的 `langchain-dashscope` 包

## 反思
- 今天的完整流程是：手写 RAG → 学 LangChain 框架 → 遇到本地性能瓶颈 → 换通义 API 跑通。每一步都是实际问题驱动，不是纯看教程
