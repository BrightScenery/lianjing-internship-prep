## 核心认知

### 1. Docker 密钥管理：镜像 vs 运行时

今天花了很多时间探索"API Key 该放哪"，经历了几次反复，最终确认：

| 方案 | 结果 |
|------|------|
| 把 `.env` 打包进镜像 (`COPY .env .env`) | 镜像可被导出反编译，Key 会泄露 |
| 构建时运行 `init_db.py` 需要 Key | 构建日志/镜像中间层可能暴露 Key |
| **运行时通过 `env_file` 注入** | ✅ Key 只在内存中，不写入镜像 |

**结论：密钥永远不写进镜像。只通过运行时环境变量注入（docker-compose 的 `env_file` 字段）。**

### 2. RAG 容器的正确架构

```
Dockerfile（构建时）                    docker-compose.yml（运行时）
  ↓                                      ↓
FROM python:3.12-slim                    env_file: .env     ← Key 注入
COPY 代码 + 数据                         volumes: 持久化向量库
                                         ports: 7860:7860
不复制 .env                              ↓
不运行 init_db.py                        容器启动 → 检查向量库
不打包密钥                                 → 存在 → 秒加载
                                           → 不存在 → 自动初始化
```

**gradio_app.py 的启动逻辑：**
- 检查 `chroma_db_gradio/` 是否存在
- 存在 → 直接加载（秒开）
- 不存在 → 读取 `data/*.md` → 分块 → 向量化 → 保存 → 加载（首次启动慢）

这比"拆成两个脚本再在 Dockerfile 的 RUN 阶段调用"更简洁，也更安全。

### 3. docker-compose 的作用

`docker run` 需要写一长串参数，docker-compose.yml 把这些写成配置：
- `build: .` — 用当前目录的 Dockerfile 构建
- `env_file: .env` — 安全注入密钥
- `volumes: ./chroma_db_gradio:/app/chroma_db_gradio` — 数据持久化
- `restart: unless-stopped` — 崩溃自动重启

一键启动：`docker compose up -d`

## 教训

不要为了"优化"牺牲安全性，也不要过度设计简单问题。首次启动慢一点（向量化）是可以接受的，这是正确的工程取舍。今天的反复改来改去让我深刻认识到：**先想清楚再动手，比写出来再改效率高得多**。
