### 学了什么

1. **ping**：测试网络连通性。`-c` 指定发包数，`-W` 指定超时秒数，`-q` 静默模式。巡检脚本里不需要连续 ping，`-c 1` 发 1 个包就够了。
2. **curl**：HTTP 请求工具。`-s` 静默模式，`-o /dev/null` 丢弃响应体，`-w "%{http_code}"` 只输出状态码，`--connect-timeout` 控制超时。`000` 表示 TCP 都连不上（不是 404，是根本没响应）。
3. **wget**：文件下载。`-O` 指定输出路径，`-q` 静默，`--timeout` 控制超时。与 curl 的区别：wget 偏向下载文件，curl 偏向发 HTTP 请求（调试 API）。
4. **ssh**：远程登录。`ssh -V` 查看版本。生产环境下用于登录 HPC 节点操作。
5. **tar 的本质**：tar 是打包（归档），不是压缩。把多个文件捆在一起，按 512 字节对齐，每个文件加 header 元信息（文件名、权限、时间戳）。真正的压缩要配合 gzip（`tar czf`）或 bzip2（`tar cjf`）。

### net-api-check.sh 的作用和意义

这个脚本模拟运维中常见的"网络巡检"场景。联旌智能的 EaaS 平台分布在多台服务器上，运维人员需要快速判断某台节点的网络是否正常：能不能 ping 通外网？Web 服务端口是否在响应？API 接口是否正常返回？能不能下载文件？脚本把零散的网络命令整合成一份"网络健康报告"，5 个检查项依次输出，有通畅/不通的明确结论，而不是扔出一大堆原始命令结果让人自己判断。虽然目前的版本还比较简单（没有告警、没有日志记录），但它是我理解"什么是自动化运维"的第二步——和昨天的 check-system.sh 一起，构成了"系统状态 + 网络状态"两个维度的巡检能力。

### 踩过的坑
- **问题**：ping 127.0.0.1 测试外网连通性
- **原因**：127.0.0.1 是本地回环地址，网线拔了也能 ping 通，它只测本机网络栈，不测外网
- **解决**：改用 8.8.8.8（Google DNS）和 baidu.com 测试外网连通性
- **问题**：变量定义成 `burl=/tmp/baidu_homepage.html`，但 `[ -f $url ]` 用的是 `$url`
- **原因**：变量名少打了 `b`，shell 中引用了未定义的变量（值为空），`[ -f "" ]` 永远为假
- **解决**：统一变量名，`if [ -f $burl ]`
- **问题**：理解 tar 为什么 5 个 52 字节的文件打包后变成 10K
- **原因**：tar 不是压缩，是归档。每个文件有 512 字节 header + 数据区按 512 字节对齐补齐，末尾还有结束标记。tar 默认 block factor 对齐到 10K
- **解决**：真正压缩需要 `tar czf`（配合 gzip）

## 💡 今日感悟
curl 和 wget 的区别以前一直没搞懂——curl 是调试工具（发请求、看响应、拿状态码），wget 是下载工具（下载文件到本地）。理解了这个区别后，什么时候用哪个就清楚了：调 API 用 curl，下载模型权重用 wget。
