## 核心理解

### Service 流量链路
```
curl localhost:30080
  → NodePort（宿主机端口）
  → kube-proxy（监听 Service 变化，写入 iptables 规则）
  → iptables（Linux 内核网络层，负载均衡选 Pod）
  → 某个 Pod 的 IP:端口
```

**关键认知**：Service 本身不是进程，只是一个 K8s 规则对象。真正转发数据包的是 kube-proxy + iptables。

### Endpoints
- Endpoints 是 Service 和 Pod 之间的"通讯录"
- kube-proxy 读的就是 Endpoints 列表
- `kubectl get endpoints` 可以看到被选中的 Pod IP
- selector 选不到 Pod → Endpoints 为空 → 连接被拒

### imagePullPolicy 与 Endpoints 验证实验
- 将 Service 的 selector 改为 `app: wrong` → Endpoints 变为 `<none>` → curl 连接被拒绝
- 改回正确的 selector → Endpoints 恢复三个 Pod IP → nginx 正常响应
- **因果验证**：selector 选不中 Pod = 没有 Endpoints = Service 是空壳

### Service 三种类型
| 类型 | 作用 | 场景 |
|------|------|------|
| ClusterIP | 仅集群内部可访问 | 微服务之间互相调用 |
| NodePort | 宿主机固定端口（30000-32767） | 外部直接访问，今天用的方式 |
| LoadBalancer | 云厂商分配公网 IP | 公有云环境 |

**k3s 的坑**：k3s 的 LoadBalancer 类型实际上是 klipper-lb 包装的 NodePort，EXTERNAL-IP 就是宿主机自己的 IP，不是真正的云负载均衡器。

### Ingress 解决的问题
- 多个服务需要记住多个 NodePort 端口（30080、30001、30002...）
- Ingress 通过 URL 路径区分服务，一个端口搞定
- 类比：公司前台，所有电话打同一个号码，前台帮你转接

### Ingress vs IngressRoute
| | 标准 Ingress | Traefik IngressRoute |
|---|---|---|
| 每条路由绑定 Middleware | 不行，只能全局绑定 | 可以，每条路由单独配 |
| 匹配语法 | 只有 Prefix/Exact | 支持正则、Header 匹配等 |
| 需要 CRD | 不需要 | 需要 |

**生产环境选择 IngressRoute**：因为 nginx 需要 StripPrefix 去掉 /nginx 前缀，Flask 不需要，标准 Ingress 无法给不同路径配不同 Middleware。

### Traefik 端口映射
```
traefik Service: 80:31304/TCP, 443:31036/TCP
```
格式：`服务端口:节点端口/协议`
- 访问 `宿主机IP:31304` → Traefik 的 80 端口
- 这是 Traefik 的入口（entrypoint: web）

### Middleware（中间件）
请求的中间处理器，在到达后端服务之前预处理请求：
- StripPrefix：去掉 URL 前缀
- 认证：没带 Token 直接 401
- 限流：超频率直接丢弃

**中间件粒度问题**：不同服务可能需要不同的 Middleware。
- nginx 需要去掉 /nginx 前缀（因为 nginx 文件在根目录 /）
- Flask 不需要去掉 /api 前缀（因为 Flask 路由自己带了 /api）
- 所以要用 IngressRoute，给每条路由单独配 Middleware

## 今日操作记录

```bash
# 查看 Service Endpoints（验证 selector 选中的 Pod）
kubectl get endpoints
kubectl get pods --show-labels

# 实验：改错 selector，观察 Endpoints 变空
# 将 service.yaml 中 selector: app: nginx 改为 app: wrong
kubectl apply -f service.yaml
kubectl get endpoints         # → <none>
curl localhost:30080          # → Connection refused

# 修复 selector，恢复正常
kubectl apply -f service.yaml
kubectl get endpoints         # → 恢复三个 Pod IP
curl localhost:30080          # → nginx 欢迎页

# 查看 Traefik 端口映射
kubectl get svc -n kube-system | grep traefik

# 创建 Middleware（去掉 /nginx 前缀）
kubectl apply -f middleware-nginx.yaml

# 创建 IngressRoute（路径分流）
kubectl apply -f ingressroute.yaml

# 验证路径路由
curl http://localhost:31304/nginx          # → nginx 欢迎页 ✅
curl http://localhost:31304/api            # → Flask 404（Flask 无 /api 路由，符合预期）
curl http://localhost:31304/api/health     # → {"status": "healthy"} ✅
```

## 今日新增文件
- `middleware-nginx.yaml` — Traefik StripPrefix 中间件
- `ingressroute.yaml` — Traefik IngressRoute 路径路由配置

## 踩坑记录
- Traefik Middleware 的 `passBody` 字段不存在，Flask 不需要 Middleware，直接不加 middlewares 字段即可
- IngressRoute 的字段名必须是复数：`services` 不是 `service`，`middlewares` 不是 `middleware`
- YAML 列表中 `- name:` 短横线和 name 之间必须有空格
- `/api/health` 正常但 `/api` 404：Flask 路由写的是 `/` 不是 `/api`，属于路由设计一致性问题，不是 Ingress 的 bug
