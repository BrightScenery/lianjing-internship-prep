### Docker 安装

选择 **Ubuntu 原生 Docker** 而非 Docker Desktop，原因：
- 更贴近真实服务器环境，实习时公司用的就是 Linux 服务器
- WSL2 环境下操作，体验和远程 SSH 到服务器一致
- 不需要图形界面，纯命令行操作

官方文档：https://docs.docker.com/engine/install/ubuntu/

### 国内镜像源配置

Docker Hub 在国内无法正常访问，需要配置镜像代理。尝试了多个源，最终可用配置：

```json
{
  "registry-mirrors": [
    "https://docker.1panel.live",
    "https://docker.1ms.run",
    "https://docker.kejilion.pro",
    "https://dockerproxy.link",
    "https://docker.cloudlayer.icu"
  ]
}
```

配置文件路径：`/etc/docker/daemon.json`

修改后必须执行：
```bash
sudo systemctl daemon-reload
sudo systemctl restart docker
```

### hello-world 验证

```
$ sudo docker run hello-world
Hello from Docker!
This message shows that your installation appears to be working correctly.
```

Docker 执行了 4 个步骤：
1. Docker 客户端联系 Docker 守护进程
2. 守护进程从 Docker Hub 拉取 hello-world 镜像
3. 守护进程用该镜像创建新容器
4. 容器的输出流回客户端，显示在终端

### 踩过的坑
- **vim 编辑只读文件无法 :wq 退出**：用 `vim /etc/docker/daemon.json` 编辑系统文件时，`:wq` 包含保存（:w），但文件是只读的，保存失败后 vim 拒绝退出。正确做法是用 `:q!` 强制退出不保存，或者先用 `sudo` 打开。后来改用 `sudo tee` 重定向写入配置文件，更简洁。
- **镜像源不稳定**：Docker Hub 国内镜像源大量关闭，需要多试几个。最终组合了 5 个源到 `daemon.json` 里，Docker 会自动逐个尝试。

### 核心概念理解

**镜像（Image）**：
- 类比：软件的"安装包"（类似 .exe 安装文件或 .iso 光盘镜像）
- 是只读模板，包含了运行应用所需的文件系统、依赖库、环境变量
- 不能直接运行，必须通过镜像创建容器才能执行
- `docker images` 查看本地已有的镜像

**容器（Container）**：
- 类比：从安装包"安装并运行"后的程序实例
- 是镜像的运行实例，有自己独立的文件系统、网络、进程空间
- 每次 `docker run` 都会创建一个新的容器实例
- 容器是临时的，删除后不影响原始镜像
- `docker ps` 查看运行中的容器，`docker ps -a` 查看所有（含已停止的）

**镜像和容器的关系**：
- 一个镜像可以创建无数个容器（就像一份 .exe 可以多次运行）
- 容器的修改不会自动保存到镜像（需要 `docker commit` 或 Dockerfile）
- 容器被删除后，镜像仍然存在

## 💡 今日感悟

**Docker 的本质是隔离**：容器不是虚拟机，而是通过 Linux 的 cgroup（资源限制）+ namespace（资源隔离）技术，让进程以为自己独占一台机器。这比虚拟机轻量得多——虚拟机要启动完整的操作系统，容器只需要隔离进程本身。理解了这一点，就能明白为什么联旌智能的 EaaS 平台要选择容器化架构：同样的硬件能跑更多服务。
