## 核心概念

### K8s 工作流程回顾
```
你提交 YAML → K8s 解析 apiVersion + kind → 创建资源对象
→ Controller 发现"期望状态 vs 实际状态不一致"
→ 创建 ReplicaSet → ReplicaSet 创建 Pod
→ 持续 reconcile：实际状态 != 期望状态 → 自动修正
```

**重要认知**：Deployment 不直接管 Pod，它通过 ReplicaSet 间接管理。这就是为什么滚动更新时能看到新旧两个 ReplicaSet 共存。

### selector 不可变
`spec.selector` 是 Deployment 创建后不可修改的字段。如果要改标签，必须 delete 重建。

## 生产级 YAML 字段详解

### 1. resources（资源限制）
```yaml
requests:  # 启动至少需要，K8s 调度依据
  cpu: "100m"    # 0.1 核心
  memory: "128Mi"
limits:    # 最大可用，超限处理
  cpu: "500m"    # 0.5 核心，超了限速
  memory: "256Mi"  # 超了 OOMKilled
```

### 2. 三种 Probe（健康检查）
| 探针 | 不通过的后果 | 适用场景 |
|------|------------|---------|
| startupProbe | 阻止其他探针运行 | 启动慢的应用 |
| livenessProbe | 杀 Pod 重启 | 死锁/假活 |
| readinessProbe | 从 Endpoints 移除 | 未准备好接客 |

**startupProbe × failureThreshold = 最大启动时间**
例：periodSeconds=10, failureThreshold=30 → 最多 5 分钟

### 3. env（环境变量）
- 直接 value：普通配置
- valueFrom.secretKeyRef：从 Secret 取敏感信息
- valueFrom.configMapKeyRef：从 ConfigMap 取配置

### 4. imagePullPolicy
- `:latest` 默认 Always，本地镜像必须显式设 IfNotPresent
- 生产环境用具体版本号 tag

### 5. rollingUpdate
- maxSurge=1：更新时最多 4 个 Pod（3+1）
- maxUnavailable=0：始终保证 3 个在线

## 实战记录

```bash
# 验证 k3s 集群
kubectl cluster-info

# 检查镜像
sudo k3s crictl images | grep flask
# → docker.io/library/flask-app:latest ✅

# 创建 Secret（解决 CreateContainerConfigError）
kubectl create secret generic app-secrets --from-literal=db-password=dummy123

# 首次 apply
kubectl apply -f full-deployment.yaml
# → deployment.apps/flask-app created

# 验证 Pod 状态
kubectl get pods -l app=flask
# → 3 个 Pod 全部 1/1 Running ✅

# 验证 Endpoints
kubectl get endpoints
# → flask-service 匹配到 Pod IP ✅
```

## 踩坑记录
1. **YAML 字段名复数**：`request` → `requests`，`limit` → `limits`，K8s 要求复数
2. **label 匹配**：Deployment 的 label 必须和 Service 的 selector 一致，否则 Endpoints 为空
3. **探针路径**：Flask 只有 `/api/health` 端点，Probe 路径要对应，不能写 `/health`
4. **env 缩进**：env 要和 probes 平级，不能嵌在 probe 里面
5. **selector 不可变**：`spec.selector` 创建后不能改，只能 delete 重建
