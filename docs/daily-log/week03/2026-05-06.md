## 核心理解

### K8s 是什么
K8s（Kubernetes）是容器编排系统。Docker 管一个容器怎么跑，K8s 管一堆容器怎么分布在多台机器上跑、挂了自动重启、流量大了自动扩容。

### K8s 层次关系
```
Deployment（部署）
  └── ReplicaSet（副本集，控制 Pod 数量）
       └── Pod（K8s 最小运行单元）
            └── Container（容器，装 nginx）
```

### 关键概念对比
| 概念 | 大白话 |
|------|--------|
| Pod | K8s 最小的运行单位，类似 Docker 的容器但更高级 |
| Deployment | "我要几个 Pod、用什么镜像"的声明 |
| Service | 给 Pod 提供固定入口，不管后面的 Pod 怎么变 |
| Node | 一台物理机或虚拟机 |
| kubectl | 指挥 K8s 的命令行工具 |

### 镜像存储隔离
k3s 使用 containerd 作为运行时，Docker 用 Docker 自己的存储。两个"仓库"不互通：
- Docker → `/var/lib/docker/`
- k3s (containerd) → `/var/lib/rancher/k3s/agent/containerd/`

所以 Docker pull 的镜像，K8s 找不到。要用 `k3s ctr images pull` 拉到 K8s 的存储里。

### 端口（Port）是什么
端口就是一个"门牌号"。Nginx 监听 80 端口 = Nginx 告诉操作系统："80 号门的请求我来处理"。没人监听端口时，访问返回 `Connection refused`。

## 踩坑记录

### 坑1：Docker Hub 国内无法访问
k3s 安装后所有系统 Pod 卡在 `ContainerCreating`，10 小时不动。原因是默认从 `docker.io` 拉镜像，国内服务器连不上。

**解决：重装 k3s，指定 `--system-default-registry`**
```bash
curl -sfL https://rancher-mirror.rancher.cn/k3s/k3s-install.sh | \
  INSTALL_K3S_MIRROR=cn \
  K3S_TOKEN=mynodetoken \
  sh -s - --system-default-registry registry.cn-hangzhou.aliyuncs.com
```

### 坑2：系统镜像 vs 用户镜像的镜像源
`--system-default-registry` 只对 k3s 系统自带镜像生效。用户自己指定的镜像（如 nginx）仍然去 docker.io 拉。

**解决：用国内替代镜像手动拉取**
```bash
sudo k3s ctr images pull registry.cn-beijing.aliyuncs.com/dotbalo/nginx:latest
```

### 坑3：minikube + kubectl 折腾一晚失败
昨晚在腾讯云服务器上装了 minikube 和 kubectl，装了一晚上各种报错没跑起来。原因是 2核4GB 的低配服务器跑 minikube 太重了，而且 minikube 默认拉镜像也走 Docker Hub。

**教训：低配服务器 + 国内环境 = 选 k3s，别碰 minikube。** k3s 是专为资源受限场景设计的轻量级 K8s，整个二进制不到 100MB，内存占用约 500MB，比 minikube 轻量得多。

## 今日操作记录

```bash
# 创建 nginx 部署
sudo k3s kubectl create deployment my-nginx \
  --image=registry.cn-beijing.aliyuncs.com/dotbalo/nginx:latest --port=80

# 扩容到 3 个副本
sudo k3s kubectl scale deployment my-nginx --replicas=3

# 暴露端口
sudo k3s kubectl expose deployment my-nginx --port=80 --type=NodePort

# 验证
sudo k3s kubectl get pods      # 3 个 Running
sudo k3s kubectl get svc       # 查看 NodePort 端口
```
