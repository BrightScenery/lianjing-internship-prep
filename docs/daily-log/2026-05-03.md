### Docker 资源限制

通过 `--cpus` 和 `--memory` 参数限制容器可以使用的资源上限：

```bash
sudo docker run -d --name stress-test \
  --cpus=0.5 \
  --memory=100m \
  nginx
```

验证结果：
```
CPU: 500000000 (= 0.5 个 CPU 核心)
Memory: 104857600 (= 100 MB)
```

**为什么需要资源限制**：
- 生产环境一台服务器通常运行多个容器
- 如果某个容器有 Bug 或被攻击，可能占满全部 CPU/内存
- 资源限制确保单个容器不会影响其他容器正常运行
- **联旌场景**：EaaS 平台多客户共享算力，必须为每个租户设置资源配额

### 非 root 用户运行

容器默认以 root 身份运行。如果容器中的代码存在漏洞被利用，攻击者将获得容器内的 root 权限。

在 Dockerfile 中创建并切换到非 root 用户：

```dockerfile
RUN groupadd -r appuser && useradd -r -g appuser -d /home/appuser -s /sbin/nologin appuser
WORKDIR /app
COPY . .
RUN chown -R appuser:appuser /app
USER appuser
```

验证非 root 用户权限受限：
```bash
sudo docker run --rm --user 1000 alpine rm -f /etc/passwd
# 报错: Permission denied
```

**核心思想**：最小权限原则——容器只需要运行应用所需的最低权限，不需要 root。

### 只读文件系统

```bash
# 根文件系统只读，写入 /tmp 会失败
sudo docker run --rm --read-only alpine touch /tmp/test

# 配合 --tmpfs 可以只让特定目录可写
sudo docker run --rm --read-only --tmpfs /tmp alpine touch /tmp/test
```

这样 `/tmp` 可写用于临时文件，但其他系统目录不可修改，防止容器被入侵后被篡改。

### 安全 Dockerfile 最佳实践总结

1. 使用精简基础镜像（slim/alpine 而非完整镜像）
2. 多阶段构建，最终镜像只包含运行时必需内容
3. 非 root 用户运行
4. `.dockerignore` 排除敏感文件（.env, .git 等）
5. 固定依赖版本，避免引入未知漏洞
6. 只读文件系统 + 必要的 tmpfs 挂载

##  今日感悟

1. **安全不是附加功能，而是设计原则**：从 Dockerfile 的第一天开始就要考虑安全，而不是写完后再补。联旌给客户交付的容器服务，安全是基本要求。
2. **资源限制是多租户的基础**：没有资源限制的多容器环境等于没有隔离，一个坏容器就能拖垮整台服务器。
3. **最小权限原则贯穿始终**：非 root 用户、只读文件系统、.dockerignore 排除敏感文件，核心思想都是"只给必需的，不多给"。
