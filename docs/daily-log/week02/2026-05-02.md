### 数据卷（Volume）vs Bind Mount

**Volume（数据卷）**：
- Docker 管理的存储区域，位于 `/var/lib/docker/volumes/`
- 不依赖宿主机的目录结构，跨平台兼容性好
- 适合需要持久化数据的场景（数据库、配置文件）
- 创建命令：`docker volume create my-vol`

**Bind Mount（绑定挂载）**：
- 直接映射宿主机的某个目录到容器内
- 依赖宿主机的路径和文件系统结构
- 适合开发时热重载代码，修改本地文件容器立即生效
- 挂载命令：`docker run -v /host/path:/container/path ...`

核心区别：Volume 由 Docker 管理，Bind Mount 由用户管理路径。

### Docker 网桥（Bridge）

- Docker 默认创建 `bridge` 网络，容器通过 NAT 与外部通信
- 默认网桥下容器间通过 IP 通信，不支持通过容器名互相访问
- 自定义网桥支持 DNS 解析，容器可以通过容器名互相访问
- 创建命令：`docker network create my-network`

### 容器间通信

在同一个自定义网络上，容器可以直接通过容器名互相访问：

```bash
docker network create app-net
docker run -d --name redis --network app-net redis
docker run -d --name flask --network app-net -e REDIS_HOST=redis flask-app
```

Flask 容器内通过 `redis:6379` 就能访问 Redis 服务。

### MySQL 持久化实验

用 Volume 挂载 MySQL 数据目录，验证删除容器后数据不丢失：

```bash
# 创建数据卷
docker volume create mysql-data

# 启动 MySQL，将数据目录挂载到 volume
docker run -d --name mysql-test \
  -v mysql-data:/var/lib/mysql \
  -e MYSQL_ROOT_PASSWORD=123456 \
  -p 3306:3306 mysql:8.0

# 进入容器创建数据库和表，插入数据
docker exec -it mysql-test mysql -uroot -p123456
CREATE DATABASE testdb;
USE testdb;
CREATE TABLE users (id INT, name VARCHAR(20));
INSERT INTO users VALUES (1, 'test');

# 删除容器
docker rm -f mysql-test

# 用同一个 volume 重新启动 MySQL
docker run -d --name mysql-test2 \
  -v mysql-data:/var/lib/mysql \
  -e MYSQL_ROOT_PASSWORD=123456 \
  -p 3306:3306 mysql:8.0

# 验证数据还在
docker exec -it mysql-test2 mysql -uroot -p123456
SELECT * FROM testdb.users;  -- 返回 (1, 'test')
```

结论：容器可以被随意删除重建，只要 volume 不变，数据就安全。这就是生产环境部署数据库的标准做法。

##  今日感悟

1. **数据持久化是容器化数据库的关键**：容器本身是 ephemeral（短暂的），数据必须外置。联旌的 EaaS 平台部署 MySQL/Redis 等服务时，底层必然依赖 Volume 做持久化。
2. **自定义网络比默认网桥更实用**：通过容器名通信比硬编码 IP 可靠得多，IP 会变但容器名不会，这在多容器编排中是基础能力。
3. **Volume vs Bind Mount 不是谁好谁坏**：生产用 Volume（安全、Docker 管理），开发用 Bind Mount（方便、实时同步），场景决定选择。
