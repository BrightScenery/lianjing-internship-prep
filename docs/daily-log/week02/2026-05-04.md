### 1. 镜像分层原理验证

用 `docker history flask-app:original` 查看了镜像的分层结构，确认了 15 层：
- 基础镜像（python:3.12-slim）本身有 9 层，约 150MB
- 自己的应用代码只有 12.3kB

**缓存机制实验**：
- 改 app.py 重新 build：Step 2-4 用了缓存，Step 5-7 重跑 ✅
- 改 requirements.txt 重新 build：Step 3-7 全部重跑（pip install 重新执行）✅

**核心规律**：Docker 缓存从上到下瀑布式失效——任何一层变了，它之后所有层都要重算。所以 Dockerfile 的写法直接影响构建速度：少变的放上面，多变的放下面。

### 3. Docker Compose 实战

完成了 Flask + Redis 计数器应用的编排：

**docker-compose.yml 结构**：
- 两个服务：flask（自建镜像）和 redis（官方镜像）
- 通过 depends_on 控制启动顺序
- 通过 environment 传递 REDIS_HOST 环境变量
- 通过 volume 持久化 Redis 数据

**数据持久化验证**：
- 计数器访问到 3
- docker-compose down 删除容器
- docker-compose up 重新启动
- 计数器从 4 继续 → 验证 volume 保留数据成功

##  今日感悟

1. **缓存是 Docker 构建速度的关键**：合理的 Dockerfile 层顺序可以节省大量构建时间。生产环境中频繁部署时，这种优化尤其重要。
2. **Compose 是理解 K8s 的前奏**：Compose 的 services/networks/volumes 概念在 K8s 中分别对应 Deployment/Pod、Network、PersistentVolumeClaim。理解单机编排是理解分布式编排的基础。
3. **数据外置是容器化数据库的铁律**：容器是短暂的，数据必须持久化到 Volume。联旌 EaaS 平台部署 MySQL/Redis 等服务时，底层必然依赖这一机制。
