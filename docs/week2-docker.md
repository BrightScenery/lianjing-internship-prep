## 学习内容

### 核心概念

| 概念 | 我的理解 |
|------|---------|
| Image（镜像） | 把应用代码、依赖、运行环境全部打包成一个分层的只读模板。每一层只存变化的部分，可以共享和复用。核心价值是"一次构建，到处运行"，消除了环境差异问题 |
| Container（容器） | 从镜像创建的运行实例，被 Linux namespace 和 cgroup 隔离在自己的"小房间"里。容器是临时的——删除后不影响镜像，但内部数据也会丢失。类比：菜谱 vs 做出来的菜 |
| Dockerfile | 用声明式语法写的构建指令文件，每一行指令生成一个镜像层。书写顺序直接影响构建速度：少变的指令放上面，多变的放下面 |
| Volume（数据卷） | Docker 提供的数据持久化机制。容器被删除后 Volume 中的数据依然保留，新容器可重新挂载。是容器化数据库和大模型数据集挂载的必要手段 |
| Port Mapping | 将容器内部端口映射到宿主机端口，格式 `-p 宿主机端口:容器端口`。两边端口号可以不同，本质是在宿主机和容器网络之间建立转发通道 |
| Docker Compose | 用一个 YAML 文件定义和管理多个容器的编排。自动配置服务间网络、启动顺序、数据卷和端口映射。单机编排工具，services/networks/volumes 概念与 K8s 一一对应 |
| 镜像分层原理 | 镜像由多个只读层堆叠而成，每层对应 Dockerfile 一条指令。构建时瀑布式缓存检查：任何一层变化，其下所有层都重做。合理编排层顺序可最大化缓存命中率 |
| 容器隔离机制（namespace + cgroup） | namespace 让容器"看不到"其他容器和宿主机的资源（进程、网络、文件系统）；cgroup 限制容器能使用的 CPU、内存等资源上限。两者结合实现轻量级隔离，无需启动完整 OS |

## 为什么学 Docker？

联旌智能的 EaaS 平台底层就是容器化架构。他们自研的容器引擎虽然技术路线可能和 Docker 不同，但核心概念（镜像、容器、隔离、资源限制）是一致的。理解 Docker 的工作方式，是理解联旌产品的基础。

## 重点场景（结合联旌实际业务）

### 场景 1：大模型数据集挂载
高校要在容器里跑训练，数据集几十 GB，不可能拷进镜像。需要 `-v /data/dataset:/app/data` 把宿主机目录挂载到容器内。这和软链接 `ln -s` 解决的是同一类问题——**让程序访问到它"本该看到"的数据，而不是复制一份**。

### 场景 2：容器日志排查
容器跑崩了，怎么看原因？`docker logs <container>` 看标准输出，`docker inspect` 看容器元信息（端口、挂载点、环境变量）。这和用 `grep error` 看日志文件是一个逻辑，只是容器没有文件系统入口，必须通过 Docker CLI 来查。

### 场景 3：环境变量与 PATH 问题
容器里输入 `python`，调用的不是你刚 `pip install` 的那个——因为 PATH 顺序不对。这和在 Linux 上遇到的 Python 环境地狱是同一个问题。需要在 Dockerfile 里用 `ENV PATH` 显式指定。

### 场景 4：端口透传
容器内服务监听 8080，外部要访问到。`-p 8080:8080` 把容器端口映射到宿主机。如果访问不了，排查步骤：`ss -tlnp` 看宿主机端口起了没、`docker ps` 看端口映射对不对、`docker exec` 进容器看服务是否在运行。
