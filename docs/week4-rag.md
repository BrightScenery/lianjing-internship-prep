## 重点场景（结合运维能力）

### 场景 1：用 Shell 处理 RAG 服务日志
100GB 日志文件中统计报错最多的模块——这正是文章第三板块的管道场景实战：
```bash
grep "Error" rag-service.log | awk '{print $5}' | sort | uniq -c | sort -nr | head -10
```
RAG 服务跑起来后，日常运维就是这套逻辑。

### 场景 2：性能排查——为什么检索很慢？
- 查 CPU Load（`htop`）：Load 高说明 CPU 瓶颈（embedding 计算慢）
- 查 iowait（`top` 里的 wa 列）：高说明磁盘 IO 瓶颈（ChromaDB 读向量文件慢）
- 查内存（`free -m`）：不够的话 ChromaDB 要频繁 swap，更慢
- 这和"会跑代码"的区别在于：**出了问题知道去哪查、知道怎么定位瓶颈**

### 场景 3：Docker 打包 RAG 应用
把整个 RAG 系统（Flask + Ollama + ChromaDB）打包成一个可交付的容器镜像。这正是联旌帮客户部署时的实际流程。
